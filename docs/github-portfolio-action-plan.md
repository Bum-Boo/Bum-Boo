- human-AI workflow tools

## Manual GitHub UI Checklist

For each featured repo, review:

- About description
- Topics
- Website/demo URL
- Social preview image
- Releases and release assets
- Pinned order
- Visibility
- README screenshot paths
- Case study link

## Release Entry Checklist

For Windows desktop repos, make the release path visible:

- Add a `Download / Release` section.
- Link to GitHub Releases.
- If no release asset exists, add a TODO naming the expected asset.
- Do not invent binaries, screenshots, users, metrics, or production claims.

Priority repos:

- `BBCC`
- `BBTxT`
- `Naver-Cafe-Posts-Downloader`
- `BIGLOADEB`

## Suggested Commit Hygiene

- Keep commits small and narrative, such as `docs: add release guidance` or `docs: clarify archive safety scope`.
- Separate docs-only portfolio work from runtime behavior changes.
- Do not commit generated build artifacts unless the repo intentionally distributes them.
- Keep screenshots and sample data sanitized.
- Document validation commands near release notes.

## External Sharing Strategy

- **GitHub profile:** concise index for the six strongest projects.
- **Portfolio site:** richer project pages with screenshots, tradeoffs, and status labels.
- **Blog/case-study posts:** explain problem, user, design goal, architecture, safety decisions, and limitations.
- **Short demo videos:** show one real workflow per repo using sanitized data.

## Do Not Overclaim

Before publishing or sharing, check that public copy does not claim:

- production readiness where the project is v0.1, mock-first, or internal-use
- broad platform support beyond what is implemented
- real OAuth, real API, or real LLM behavior where only placeholders exist
- access bypass, scraping beyond allowed scope, credential validation, or exploit execution
- real users, customers, metrics, releases, screenshots, or demo assets that do not exist
- privacy guarantees beyond the documented local storage and safety boundaries
